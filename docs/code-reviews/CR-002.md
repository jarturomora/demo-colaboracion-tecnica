## 🔍 Archivo Revisado: `src/app.js`

### Comentario #1 – Líneas 10-25 (Configuración de Helmet)

**Tipo:** ⚠️ Observación  
**Archivo:** `src/app.js`  
**Líneas:** 10-25

```javascript
app.use(
  helmet({
    contentSecurityPolicy: {
      directives: {
        defaultSrc: ["'self'"],
        styleSrc: ["'self'", "'unsafe-inline'"],
        scriptSrc: ["'self'"],
      },
    },
  })
);
```

**Comentario de @laura-torres:**
> El directivo `styleSrc` incluye `'unsafe-inline'`, lo que permite estilos CSS en línea directamente en el HTML. Esto relaja la protección contra inyección de contenido. ¿Es realmente necesario? Revisé el código HTML del cliente y no veo estilos en línea en los archivos `public/`. La plantilla HTML del correo electrónico usa estilos en línea, pero esa se genera en el servidor y nunca es servida al navegador, por lo que no está sujeta al CSP.

**Respuesta de @carlos-mendez:**
> Tienes toda la razón. El `'unsafe-inline'` era un remanente de una versión anterior donde teníamos estilos en línea en el HTML. Eliminé esa directiva; el CSP ahora es más estricto:
> ```javascript
> styleSrc: ["'self'"],
> ```
> Verifiqué que el formulario sigue funcionando correctamente sin esa directiva.

**Resolución:** ✅ Resuelto – CSP mejorado eliminando `'unsafe-inline'`.

### Comentario #2 – Líneas 27-38 (Rate Limiting)

**Tipo:** ⚠️ Observación  
**Archivo:** `src/app.js`  
**Líneas:** 27-38

```javascript
const limiter = rateLimit({
  windowMs: 15 * 60 * 1000,
  max: 5,
  message: {
    success: false,
    mensaje: 'Demasiadas solicitudes.',
  },
});
```

**Comentario de @laura-torres:**
> Dos observaciones:
>
> 1. El límite de **5 peticiones en 15 minutos** parece muy restrictivo. Un usuario con problemas de red puede necesitar varios intentos. Sugiero aumentarlo a 10.
>
> 2. El mensaje de error actual ("Demasiadas solicitudes.") no le dice al usuario cuánto tiempo debe esperar. Añadir información sobre el tiempo de espera mejoraría la experiencia de usuario.

**Respuesta de @carlos-mendez:**
> Buena observación en ambos puntos. Ajusté la configuración:
> ```javascript
> max: 10,
> message: {
>   success: false,
>   mensaje: 'Demasiadas solicitudes. Por favor, intenta de nuevo en 15 minutos.',
> },
> ```
> También configuré `standardHeaders: true` para que la respuesta incluya el header `Retry-After` con el tiempo de espera exacto.

**Resolución:** ✅ Resuelto – Límite ajustado a 10 y mensaje mejorado.

### Comentario #3 – Líneas 95-100 (Manejo de rutas 404)

**Tipo:** 💡 Sugerencia  
**Archivo:** `src/app.js`  
**Líneas:** 95-100

```javascript
app.use((req, res) => {
  res.status(404).sendFile(path.join(__dirname, '../public/404.html'));
});
```

**Comentario de @laura-torres:**
> Me parece bien el manejo de 404. Una pequeña sugerencia: sería útil registrar en los logs las rutas no encontradas (especialmente las rutas de API) para detectar posibles errores de configuración o intentos de acceso a rutas inexistentes.

**Respuesta de @carlos-mendez:**
> Buen punto. Agregué un log para las rutas de API que devuelven 404:
> ```javascript
> app.use((req, res) => {
>   if (req.path.startsWith('/api/')) {
>     console.warn(`[404] Ruta de API no encontrada: ${req.method} ${req.path}`);
>   }
>   res.status(404).sendFile(path.join(__dirname, '../public/404.html'));
> });
> ```

**Resolución:** ✅ Resuelto – Logs agregados para rutas de API no encontradas.

### Comentario #5 – Atributo `novalidate` en el formulario

**Tipo:** ❓ Pregunta  
**Archivo:** `public/index.html`

```html
<form id="formularioContacto" class="formulario" novalidate>
```

**Comentario de @laura-torres:**
> El atributo `novalidate` deshabilita la validación nativa del navegador. Asumo que esto es intencional para usar nuestra validación personalizada con JavaScript. ¿Qué pasa si JavaScript no está disponible o el usuario lo tiene deshabilitado? ¿El formulario llegaría al servidor sin validación del cliente?

**Respuesta de @carlos-mendez:**
> Correcto, `novalidate` es intencional para usar nuestra validación personalizada que tiene mensajes en español y mejor integración con el diseño. Si JavaScript está deshabilitado, la validación del cliente no funcionará, pero la validación del servidor con `express-validator` sigue activa y el servidor devuelve los errores correctamente. El formulario es funcional en ambos escenarios; solo la experiencia de usuario es mejor con JavaScript.

**Resolución:** ✅ Aclarado – Comportamiento correcto confirmado.

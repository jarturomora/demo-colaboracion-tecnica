## 🔒 Medidas de Seguridad Implementadas

### Helmet – Cabeceras HTTP de seguridad

```javascript
app.use(
  helmet({
    contentSecurityPolicy: {
      directives: {
        defaultSrc: ["'self'"],
        styleSrc: ["'self'", "'unsafe-inline'"],
        scriptSrc: ["'self'"],
      },
    },
  })
);
```

Cabeceras configuradas:
- `Content-Security-Policy` – Previene inyección de contenido.
- `X-Frame-Options: DENY` – Previene clickjacking.
- `X-Content-Type-Options: nosniff` – Previene MIME sniffing.
- `Strict-Transport-Security` – Fuerza HTTPS.

### Rate Limiting – Limitación de peticiones

```javascript
const limiter = rateLimit({
  windowMs: 15 * 60 * 1000, // 15 minutos
  max: 10,                   // máximo 10 envíos
});
```

El límite solo aplica a la ruta `POST /api/contacto`, no a las rutas de páginas estáticas.

## 💬 Comentarios del Revisor

**@laura-torres:** Me parece bien la implementación de seguridad. Tengo dos observaciones:

1. El Content Security Policy usa `'unsafe-inline'` para los estilos CSS. ¿Podríamos refactorizar el CSS para eliminar los estilos en línea y usar `'self'` únicamente?

2. El límite de 10 peticiones en 15 minutos podría ser muy restrictivo si un usuario tiene problemas técnicos y necesita reintentar varias veces. ¿Consideraste un límite más alto o un mensaje más informativo?

**@carlos-mendez:** 

1. Tienes razón con el `unsafe-inline`. La plantilla del correo HTML usa estilos en línea, pero eso es solo para el correo, no para el frontend. Revisé el código y resulta que el `unsafe-inline` no era necesario para los archivos estáticos. Lo eliminé en este commit. El HTML del correo se genera en el servidor y no pasa por el CSP del cliente.

2. Subí el límite a 10 envíos y mejoré el mensaje de error para que indique cuánto tiempo debe esperar el usuario. Creo que es un balance razonable; si alguien está teniendo problemas técnicos, puede contactarse por teléfono o redes sociales.

**@laura-torres:** Ambas correcciones están perfectas. Aprobado para fusión.
